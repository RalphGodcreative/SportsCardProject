# Implementation Plan: Storage Table

## Summary

A new `storages` table lets each user define named physical storage locations (e.g. "Box 1", "Binder A", "Closet Shelf") and assign cards to one. Root entity, scoped per user like `Card`/`Transaction`/`SearchKeyword` (see `data-ownership.md`) — a `storages` row has `id`, `name`, `user_id`. Relationship is **one storage → many cards**: `cards` gains a nullable `storage_id` FK. A card doesn't have to be assigned to a storage, and a storage can hold any number of cards.

Deleting a storage does not delete its cards — `storage_id` is set back to `NULL` (`ON DELETE SET NULL`), so cards just become unassigned again.

---

## 1. `Storage.java` — New Entity

```java
package RGcards.SportsCardProject.entity;

import jakarta.persistence.*;
import lombok.AllArgsConstructor;
import lombok.Data;
import lombok.NoArgsConstructor;

@Data
@NoArgsConstructor
@AllArgsConstructor
@Entity
@Table(name = "storages")
public class Storage {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private long id;

    private String name;

    @Column(name = "user_id")
    private Long userId;
}
```

## 2. `V7__add_storages_table.sql` — New Flyway Migration

```sql
CREATE TABLE storages (
    id      BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name    VARCHAR(255) NOT NULL,
    user_id BIGINT       NOT NULL,
    CONSTRAINT fk_storages_user FOREIGN KEY (user_id) REFERENCES users(id)
);

ALTER TABLE cards ADD COLUMN storage_id BIGINT;
ALTER TABLE cards ADD CONSTRAINT fk_cards_storage
    FOREIGN KEY (storage_id) REFERENCES storages(id) ON DELETE SET NULL;
```

## 6. `StorageService.java` — New Service

```java
package RGcards.SportsCardProject.service;

import RGcards.SportsCardProject.dao.StorageRepository;
import RGcards.SportsCardProject.entity.Storage;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;

import java.util.List;
import java.util.Optional;

@Service
@RequiredArgsConstructor
public class StorageService {

    private final StorageRepository storageRepository;

    public List<Storage> findAllForUser(Long userId) {
        return storageRepository.findByUserIdOrderByNameAsc(userId);
    }

    public Storage create(String name, Long userId) {
        Storage storage = new Storage();
        storage.setName(name);
        storage.setUserId(userId);
        return storageRepository.save(storage);
    }

    public Optional<Storage> findByIdAndUserId(long id, Long userId) {
        return storageRepository.findByIdAndUserId(id, userId);
    }

    public void delete(long id, Long userId) {
        storageRepository.findByIdAndUserId(id, userId)
                .ifPresent(storage -> storageRepository.deleteById(storage.getId()));
    }

    public Optional<Storage> rename(long id, String name, Long userId) {
        return storageRepository.findByIdAndUserId(id, userId)
                .map(storage -> {
                    storage.setName(name);
                    return storageRepository.save(storage);
                });
    }
}
```

`delete` and `rename` both re-fetch scoped by `userId` first so one user can't touch another user's storage by guessing an id — same guard shape as `findByIdAndUserId` elsewhere in the app.

## 7. `StorageController.java` — New Controller (Page + Actions)

Rather than a JSON REST API, this mirrors `CrawlerController`'s shape: a `@Controller` that serves its own Thymeleaf page at `GET /storage`, plus `@ResponseBody` action endpoints the page's JS calls via `fetch` (`PUT` to add, `DELETE` to delete, `PUT` to rename) — same verbs/pattern as `/crawler/add` and `/crawler/delete`.

```java
package RGcards.SportsCardProject.controller;

import RGcards.SportsCardProject.entity.Storage;
import RGcards.SportsCardProject.entity.User;
import RGcards.SportsCardProject.service.StorageService;
import lombok.RequiredArgsConstructor;
import org.springframework.security.core.annotation.AuthenticationPrincipal;
import org.springframework.stereotype.Controller;
import org.springframework.ui.Model;
import org.springframework.web.bind.annotation.*;

import java.util.List;

@Controller
@RequestMapping("/storage")
@RequiredArgsConstructor
public class StorageController {

    private final StorageService storageService;

    @GetMapping
    public String storageHome(Model model, @AuthenticationPrincipal User currentUser) {
        List<Storage> storages = storageService.findAllForUser(currentUser.getId());
        model.addAttribute("storages", storages);
        return "storage/list";
    }

    @PutMapping("/add")
    @ResponseBody
    public Boolean addStorage(@RequestParam(name = "name") String name,
                               @AuthenticationPrincipal User currentUser) {
        return storageService.create(name, currentUser.getId()) != null;
    }

    @PutMapping("/rename")
    @ResponseBody
    public Boolean renameStorage(@RequestParam(name = "storageId") long storageId,
                                  @RequestParam(name = "name") String name,
                                  @AuthenticationPrincipal User currentUser) {
        return storageService.rename(storageId, name, currentUser.getId()).isPresent();
    }

    @DeleteMapping("/delete")
    @ResponseBody
    public Boolean deleteStorage(@RequestParam(name = "storageId") long storageId,
                                  @AuthenticationPrincipal User currentUser) {
        storageService.delete(storageId, currentUser.getId());
        return true;
    }
}
```

## 10. Storage Dropdown on `addTransactionPage.html` (Multi-Card, JS-Built Rows)

This page is more involved: each card is a `<tr class="card-data">` built by string concatenation in the `addCard()` JS function (for rows added after page load), plus one static row already in the HTML for the first card. Fields are read back out by class name in `getCardList()` and pushed into the `cards` array of the `TransactionWithCard` payload posted to `/card/saveTransaction`.

Plan:

- `CardController.addTransaction` needs to load storages the same way as `addNewCard`, and pass them to the model:

  ```java
  @GetMapping("/addTransaction")
  public String addTransaction(Model model, @AuthenticationPrincipal User currentUser) {
      model.addAttribute("storages", storageService.findAllForUser(currentUser.getId()));
      return "addTransactionPage";
  }
  ```

- Since new rows are assembled in JS (not re-rendered by Thymeleaf per row), the storages list needs to reach JS as data, not just HTML. Serialize it once at page load via Thymeleaf's inline-JS syntax and build a reusable options-HTML string from it:

  ```html
  <script th:inline="javascript">
    var storages = /*[[${storages}]]*/ [];
    var storageOptionsHtml = '<option value="">-- none --</option>' +
      storages.map(function (s) {
        return '<option value="' + s.id + '">' + s.name + '</option>';
      }).join('');
  </script>
  ```

- Add a `<select class="storage">` cell to **both** places a row is defined:
  1. The static first `<tr class="card-data">` already in the HTML — render its options from `storageOptionsHtml` on `$(document).ready` (since Thymeleaf can't loop into a row that's meant to also be a JS template).
  2. The template string inside `addCard()` — interpolate `storageOptionsHtml` directly into the new `<td><select class="storage">...</select></td>` when building `newCard`.
- `getCardList()` gains a line reading the new field, matching the `Card` entity's `storageId` property name (Jackson maps this straight onto `Card.storageId` when the JSON body is deserialized server-side — no controller change needed here beyond what's already in `Card.java`):

  ```js
  storageId: $(this).find(".storage").val() || null,
  ```

- `.clear` handler: reset the row's `.storage` select back to the empty/none option.
- `.copy` / `.paste` handlers: extend `clipboardCard` with a `storage` property and copy/paste the selected value, same as every other field.

No change needed to `TransactionWithCard` / `Transaction` — only `Card` gains the field, and that's already covered by section 3.

## 12. Storage Column on the Card View Tables (Editable via `<select>`)

A `storage` column on the tables that list card data — `allCard.html`, `pagingAllCard.html`, and the `#card-table` read view inside `transactionList.html` — following the same inline `show`/`update-element` pattern every other editable field already uses there (toggled by `.modify`/`.cancel`, collected in `.save`, POSTed to `/card/updateCard`).

Since `Card` only stores a `storageId`, not a name, the templates need a way to resolve id → name for the read-only `show` state, and a set of options for the `<select>`. One `Map<Long, String>` (id → name) covers both — Thymeleaf can iterate a map's entries directly for the `<option>`s, so there's no need for a separate `storages` list on top of it. Add a small helper to `StorageService`:

```java
// StorageService.java — also needs `import java.util.Map;` and `import java.util.stream.Collectors;`
// added to the import list from section 6.
public Map<Long, String> findNameMapForUser(Long userId) {
    return findAllForUser(userId).stream()
            .collect(Collectors.toMap(Storage::getId, Storage::getName));
}
```

- Controllers — add `model.addAttribute("storageNames", storageService.findNameMapForUser(currentUser.getId()))` to:
  - `CardController.allCards` (`allCard.html`)
  - `CardController.allCardsByPage` (`pagingAllCard.html`)
  - `TransactionController.showTransactionById` (renders `transactionList.html`) — this class needs `StorageService` injected too (it currently only has `CardService`), same field-injection style as `CardController`.
  - `CardController.getTransactionOfCard` (also renders `transactionList.html`) — **prerequisite:** this method's current signature is `getTransactionOfCard(Model model, @PathVariable String cardId)`, with no `@AuthenticationPrincipal User currentUser` param at all, so `currentUser.getId()` isn't available yet. Add the param first:

    ```java
    @GetMapping("/cardTransaction/{cardId}")
    public String getTransactionOfCard(Model model, @PathVariable String cardId,
                                        @AuthenticationPrincipal User currentUser) {
        ...
        model.addAttribute("storageNames", storageService.findNameMapForUser(currentUser.getId()));
        return "transactionList";
    }
    ```

    (Side note, not part of this feature: unlike `TransactionController.showTransactionById`, this method has no ownership check on the transaction/cards it loads — pre-existing gap, unrelated to storage.)

- Templates — add a `storage` column next to the others (e.g. after `value`, before `note`), same shape in all three:

  ```html
  <th>storage</th>
  ...
  <td>
    <a class="show show-storage" th:text="${card.storageId != null ? storageNames.get(card.storageId) : ''}"></a>
    <select class="update-element storage">
      <option value="">-- none --</option>
      <option th:each="entry : ${storageNames}" th:value="${entry.key}" th:text="${entry.value}"
              th:selected="${card.storageId != null and card.storageId == entry.key}"></option>
    </select>
  </td>
  ```

- JS — add one line to each template's existing `.save` handler, in the `cardData` object right after `value`:

  ```js
  storageId: row.find(".storage").val() || null,
  ```

  No other JS changes needed: `.modify`/`.cancel` already toggle `.show`/`.update-element` generically by class, not by field name, so the existing handlers pick this column up for free.

**⚠ Required, not optional: `transactionList.html`'s `.existing-cards-table` must at least preserve `storageId`.** This is a separate table from `#card-table` above — it's the JS-built bulk-edit rows shown while a transaction is in edit mode (`.tran-edit-butt`). Its `.tran-save-butt` handler collects every row in `.existing-cards-table` (not just changed ones) into `existingCardUpdates`, and posts each one to `/card/updateCard` — the same full-entity-replace endpoint used everywhere else. Since `/card/updateCard` overwrites every column with whatever the payload contains (that's why every other field is already threaded through every `.save`/`existingCardUpdates` call site), **any existing-card row resubmitted here without a `storageId` will have its storage silently reset to `NULL`** — meaning simply editing a transaction's date and clicking save, without touching a single card, would wipe the storage assignment of every card in that transaction, once storage exists at all.

The minimal fix (no dropdown needed, just data preservation) is a hidden field on each server-rendered existing-card row:

```html
<!-- inside the th:each="card : ${transactionWithCard.cards}" row in .existing-cards-table -->
<input type="hidden" class="storage-id" th:value="${card.storageId}" />
```

and one line added to the `existingCardUpdates` object in `.tran-save-butt`:

```js
storageId: row.find(".storage-id").val() || null,
```

A full `<select>` here (matching section 10's options-HTML-splice approach, so the user could actually *change* storage from this table) is a nice-to-have on top of that — but the hidden-field pass-through is not optional once any of sections 12-15 ship, or transaction edits will quietly erase storage data elsewhere in the app.

## 13. Storage on `searchCardPage.html` (Search Criterion + Result Column)

## 15. Storage on `sellTransactionPage.html` (Search-to-Sell Flow)

Another page missed in the original pass: `/card/sellTransaction` has its own search criteria row and its own JS-templated results table (built in a `$("#search-butt").on("click", ...)` handler, calling the same `/cardRest/searchCard` endpoint as `searchCardPage.html`). Unlike the other tables, though, these result rows have **no edit affordance at all** — no `.modify`/`.save`, just an "add" button that pushes the card onto a `cardIds` array for the sale. So storage here is display-only, same treatment as `note` already gets on this page (a plain `<a class=" note">` with no editable counterpart) — not a `<select>`.

Showing it is more than cosmetic here: knowing which box/binder a card lives in is exactly what you need before pulling it to ship a sale.

- Controller — `CardController.sellTransaction` currently returns the view with no model:

  ```java
  @GetMapping("/sellTransaction")
  public String sellTransaction(Model model) {
      return "sellTransactionPage";
  }
  ```

  Needs the same `storageNames` map, user-aware like the other GETs:

  ```java
  @GetMapping("/sellTransaction")
  public String sellTransaction(Model model, @AuthenticationPrincipal User currentUser) {
      model.addAttribute("storageNames", storageService.findNameMapForUser(currentUser.getId()));
      return "sellTransactionPage";
  }
  ```

- Search criteria row — same `<select id="storage">` as section 13b, and the same one-line addition to this page's own `formData` object in the `$("#search-butt").on("click", ...)` handler:

  ```js
  storageId: $("#storage").val() || null,
  ```

- Results table — serialize `storageNames` into JS the same way as section 13c, then add a plain (non-editable) `storage` column matching this page's existing style for `note` (note the page's own quirky `class=" fieldname"` — leading space — convention, kept here for consistency with its neighbors):

  ```html
  <script th:inline="javascript">
    var storageNames = /*[[${storageNames}]]*/ {};
  </script>
  ```

  ```js
  <td><a class=" storage">${card.storageId != null && storageNames[card.storageId] ? storageNames[card.storageId] : ''}</a></td>
  ```

- `.add-card` handler — this page also builds a short `createCardInfo(cardData)` summary string shown in the "included cards" list once a card is added to the sale. Worth carrying storage through there too, since that's the point in the flow where the seller is about to go find the physical card:

  ```js
  // add to the cardData object built in the .add-card handler:
  storage: row.find(".storage").text(),
  ```

  ```js
  // add to createCardInfo, alongside the other "if (cardData.x != "")" blocks:
  if (cardData.storage != "") {
    info += "[" + cardData.storage + "]";
    info += " ";
  }
  ```
